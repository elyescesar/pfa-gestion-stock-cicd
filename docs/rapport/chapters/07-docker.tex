\chapter{Conteneurisation Docker}
\label{ch:docker}

\section{Rôle de Docker dans le projet}

Docker matérialise l'environnement d'exécution : mêmes versions de runtime, mêmes dépendances système, isolation des services. Le fichier \texttt{docker/compose.dev.yml} couvre le développement local (dont Adminer pour visualiser PostgreSQL) ; les images construites en CI alimentent minikube et, optionnellement, GHCR pour un VPS.

\subsection{Variables d'environnement API}

\begin{itemize}
  \item \texttt{DATABASE\_URL=postgresql://stock:stock@postgres:5432/stock\_db} ;
  \item \texttt{SECRET\_JWT}, \texttt{CORS\_ORIGINS} incluant \texttt{http://localhost:3000}.
\end{itemize}

\subsection{Script de démarrage}

\texttt{scripts/demarrer-dev.sh} copie \texttt{.env.example} vers \texttt{.env} si nécessaire, lance \texttt{compose up --build}.

\section{Dockerfile backend}

Image basée sur \texttt{python:3.12-slim} : dépendances pip, \texttt{PYTHONPATH=/app}, commande uvicorn sur le port~8000.

\section{Dockerfile frontend}

Build multi-stage : \texttt{VITE\_API\_URL} figé au build. En dev Compose : \texttt{http://localhost:8000} ; pour minikube : \texttt{http://api.pfa.test} (script \texttt{demarrer-cluster-minikube.sh}).

\section{Images et CI}

Le job \texttt{docker} de la CI valide les builds. Le job \texttt{kubernetes} charge les images dans minikube. Le workflow optionnel \texttt{deploy.yml} pousse vers GHCR pour un VPS.

\section{Réseau et volumes}

Réseau \texttt{pfa} (bridge) ; volume \texttt{donnees\_pg} pour persistance PostgreSQL en dev.

\section{Bonnes pratiques}

Images \texttt{alpine}/\texttt{slim}, secrets hors Dockerfile, builds validés en CI.
