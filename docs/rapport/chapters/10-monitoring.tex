\chapter{Observabilité — Prometheus et Grafana}
\label{ch:monitoring}

\section{Objectifs}

L'observabilité vise à répondre, en exploitation :
\begin{itemize}
  \item l'API reçoit-elle du trafic ? à quel rythme ?
  \item quelle est la latence (p50, p95, p99) ?
  \item y a-t-il des erreurs 5xx ?
  \item quelles routes sont les plus sollicitées ?
\end{itemize}

StockFlow adopte la stack \textbf{Prometheus} (collecte) + \textbf{Grafana} (visualisation), standard dans l'écosystème Kubernetes.

\section{Instrumentation applicative}

Le package \texttt{prometheus-fastapi-instrumentator} expose automatiquement sur \texttt{GET /metrics} :
\begin{itemize}
  \item compteur \texttt{http\_requests\_total} (labels method, handler, status) ;
  \item histogrammes de durée de requête ;
  \item métriques en vol (\texttt{inprogress}).
\end{itemize}

Aucune modification manuelle par route n'est nécessaire pour les métriques HTTP de base.

\section{kube-prometheus-stack}

\subsection{Démo locale (minikube)}

Le script \texttt{demarrer-cluster-minikube.sh} installe le chart \texttt{kube-prometheus-stack} avec \texttt{monitoring/kube-prometheus-values-minikube.yaml} :
\begin{itemize}
  \item ressources réduites (Alertmanager désactivé, rétention courte) ;
  \item Grafana accessible via ingress \texttt{grafana.test} ou \texttt{minikube service} ;
  \item sidecar dashboards — surveille les ConfigMaps \texttt{grafana\_dashboard: "1"}.
\end{itemize}

\subsection{Production optionnelle (VPS)}

Terraform peut installer le même chart avec \texttt{monitoring/kube-prometheus-values.yaml} et TLS sur \texttt{grafana.pfa.elyes.dev}.

\section{ServiceMonitor}

Le CRD \texttt{ServiceMonitor} (Prometheus Operator) décrit comment scraper une cible. Le chart \texttt{pfa-stock} crée :

\begin{lstlisting}[style=yaml,caption={Extrait ServiceMonitor (conceptuel)}]
spec:
  selector:
    matchLabels:
      app: api
  endpoints:
    - port: http
      path: /metrics
      interval: 30s
\end{lstlisting}

Le label \texttt{release: kube-prometheus-stack} associe ce monitor à l'instance Prometheus déployée par le chart homonyme.

\section{Dashboard Grafana}

Le fichier \texttt{monitoring/dashboards/pfa-stock-api.json} définit le tableau de bord \emph{PFA Stock — API FastAPI} avec panneaux :
\begin{itemize}
  \item débit de requêtes ;
  \item erreurs 5xx ;
  \item latences p50 / p95 / p99 ;
  \item répartition par route et par code HTTP ;
  \item requêtes en cours.
\end{itemize}

Le manifeste ConfigMap est appliqué par le script minikube (ou Terraform sur VPS) ; le sidecar Grafana charge le dashboard — versionné comme le code.

\section{Accès opérationnel}

\subsection{Démo soutenance (minikube)}

\begin{itemize}
  \item Grafana : \texttt{http://grafana.test} (entrée \texttt{/etc/hosts} vers \texttt{minikube ip}) ou port-forward affiché par \texttt{make cluster} ;
  \item identifiants : admin / mot de passe défini dans \texttt{kube-prometheus-values-minikube.yaml}.
\end{itemize}

\subsection{Production optionnelle}

Grafana sur \texttt{https://grafana.pfa.elyes.dev} après \texttt{terraform apply} sur VPS.

\section{Mode développement Compose}

En \texttt{make dev}, Prometheus/Grafana ne sont pas déployés. Les métriques restent accessibles sur \texttt{http://localhost:8000/metrics}. Adminer (\texttt{http://localhost:8080}) couvre la visualisation PostgreSQL sur ce chemin.

\section{Alerting}

Alertmanager est inclus dans kube-prometheus-stack mais désactivé en minikube pour économiser la RAM. Les règles d'alerte custom (stock critique, API down prolongé) ne sont pas configurées — extension naturelle pour une phase ultérieure.
