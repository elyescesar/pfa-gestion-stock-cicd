\chapter{Intégration continue et déploiement — GitHub Actions}
\label{ch:cicd}

\section{Principes}

Deux workflows :
\begin{itemize}
  \item \texttt{ci.yml} — validation complète (tests, build, Terraform, Helm, \textbf{minikube}) ;
  \item \texttt{deploy.yml} — déploiement VPS \textbf{optionnel} (\texttt{workflow\_dispatch} uniquement).
\end{itemize}

\section{Déclencheurs}

\begin{itemize}
  \item CI : \texttt{push} sur \texttt{main}/\texttt{develop}, \texttt{pull\_request} vers \texttt{main} ;
  \item CD : \texttt{workflow\_dispatch} seulement (pas de déploiement auto sur push).
\end{itemize}

\section{Pipeline CI}

Six jobs : \texttt{backend}, \texttt{frontend}, \texttt{docker}, \texttt{terraform}, \texttt{helm}, \texttt{kubernetes}.

Le job \texttt{kubernetes} :
\begin{enumerate}
  \item démarre minikube (driver Docker) ;
  \item build et \texttt{minikube image load} des images API/Web ;
  \item active l'addon ingress ;
  \item \texttt{helm upgrade --install pfa-stock} avec \texttt{values-minikube.yaml} ;
  \item smoke test in-cluster : \texttt{curl http://api:8000/health}.
\end{enumerate}

\section{Pipeline CD optionnel (\texttt{deploy.yml})}

Déclenchement manuel : build GHCR, \texttt{helm upgrade} sur VPS (\texttt{values-prod.yaml}), smoke in-cluster et HTTPS public (\texttt{continue-on-error}).

\section{Gouvernance}

Les PR doivent passer la CI avant merge. La validation PFA repose sur la CI et la démo locale (\texttt{docs/demo-soutenance.md}), sans serveur de production obligatoire.
