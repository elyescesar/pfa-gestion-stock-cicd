\chapter{Intégration continue et déploiement — GitHub Actions}
\label{ch:cicd}

\section{Principes}

Deux workflows complémentaires :
\begin{itemize}
  \item \texttt{ci.yml} — validation (tests, lint, build, Terraform, Helm) ;
  \item \texttt{deploy.yml} — déploiement production sur push \texttt{main} (images GHCR + \texttt{helm upgrade}).
\end{itemize}

\section{Déclencheurs}

\begin{itemize}
  \item CI : \texttt{push} sur \texttt{main}/\texttt{develop}, \texttt{pull\_request} vers \texttt{main} ;
  \item CD : \texttt{push} sur \texttt{main}, \texttt{workflow\_dispatch}.
\end{itemize}

\section{Pipeline CI}

Cinq jobs parallèles : \texttt{backend}, \texttt{frontend}, \texttt{docker}, \texttt{terraform}, \texttt{helm}. Le job \texttt{kubernetes} (kind) a été retiré — la validation Helm utilise \texttt{values-prod.yaml}.

\section{Pipeline CD (\texttt{deploy.yml})}

\begin{enumerate}
  \item Build et push \texttt{ghcr.io/<owner>/pfa-stock-api} et \texttt{pfa-stock-web} (tag = SHA du commit) ;
  \item \texttt{VITE\_API\_URL=https://api.pfa.elyes.dev} pour le build frontend ;
  \item Configuration du kubeconfig depuis le secret \texttt{KUBECONFIG} ;
  \item \texttt{helm upgrade --install} avec \texttt{values-prod.yaml} et secrets (\texttt{SECRET\_JWT}, \texttt{POSTGRES\_PASSWORD}) ;
  \item Smoke test : \texttt{curl https://api.pfa.elyes.dev/health}.
\end{enumerate}

\section{Gouvernance}

Les PR doivent passer la CI avant merge. Le déploiement production est déclenché uniquement après intégration sur \texttt{main}, limitant les déploiements accidentels.
